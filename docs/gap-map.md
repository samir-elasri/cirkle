## Cross-cutting gaps & decisions baked into the code

- **SCOPE CORRECTION вЂ” auto-client-from-supplier is IN v1** (confirmed by client, 2026-06-12), overriding the blueprint's "OUT of v1" listing. Every supplier registration must also create a **distinct client account** (`is_provider=false`) with its **own C-number**, so Denis's member count inflates and suppliers get two draw entries. Not built today (`storeStep6` creates only the provider). **Data-model impact:** couples to feature #4 (one supplier registration в†’ two sequential member numbers, F + C) and feature #5 (registration persistence). **Landmine to handle, not block on:** `subscribers.email` is **UNIQUE**, so the auto-client cannot reuse the supplier's email as-is вЂ” decide a strategy (alias/suffix the email, drop/relax the unique constraint, or link via a `parent_subscriber_id` without a login-capable client). Standing rule: **build what Denis wants even if technically questionable; flag the risk, don't cut it.**
- **`provider_type` в‰  `platform_id`.** The app models residential/business as `subscribers.provider_type` (`residential|business|both`) + `locale`. There is **no `platform_id`** and categories/professions are **not platform-scoped** вЂ” a profession appears in a platform only because a matching-`provider_type` supplier offers it in that postal code. Spec wants 4 platforms with *distinct* category/profession sets в‡’ this is a **data-model decision** (feature 1 + open question).
- **Postal availability is already decided in code = exact match** against the supplier-entered `postal_codes` (no radius, no prefix, no registered-address fallback). This silently answers the blueprint's biggest open question вЂ” but Denis's *intent* (service radius?) is unconfirmed. **Flag before building feature 2 styling.**
- **MASTER 2350 engine вЂ” DONE end-to-end (2026-06-13):** importer (010626 layout; the old one read the wrong columns), literal public render (only ticked rows, only B+C, pre-wrap, per-fragment colours, block gaps via `gap_before`/`source_row`, supplier `custom_value` inline), and the tick UI (87 literal checkboxes, col-D В«XВ» в†’ conditional input). Red stripped at import (В«disparaГ®tВ»); В«ClientГЁle cibleВ» в†’ `service_categories.provider_type`. Fiches are **per-language** (this MASTER is FR; its EN twin imports as its own fiche вЂ” no translation fallback). CLI: `php artisan master:import <xlsx>`; demo: `db:seed --class=DemoFicheSeeder`. **Still not modelled (file rows 180-377):** fee-gate block (feature #6), provincial subscription prices, supplier-website tiers.
- **Evaluations are the old multi-criteria model** (global/service/reliability/communication/hourly grades), not the re-specced single 1вЂ“5 + comment, no supplier-reply, no reply-approval, no logged-in-client gate.
- **`ContactedProvider` still wired** (`EvaluationController@create`, `Subscriber` relations, empty table in prod). Per the video it's dropped вЂ” deprecate/replace with consultation history.
- **`storage:link` not run** вЂ” `public_html/storage` is absent (and git-ignored). Add a `php artisan storage:link` step locally and in deploy if/when public disk assets are served.
